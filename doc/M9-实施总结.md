## 需要用户执行的步骤

### 步骤 1: 运行数据库迁移

访问初始化端点以应用数据库迁移：

```
GET https://your-domain.com/init/{your-jwt-secret}
```

**重要**: 将 `{your-jwt-secret}` 替换为您的实际 JWT 密钥（在 `wrangler.toml` 中配置的 `jwt_secret`）

这将执行 `v2_4DB()` 迁移，创建必要的表和字段。

### 步骤 2: 为用户启用 API Key 权限

有两种方式为用户启用 API Key 功能：

#### 方式 A: 通过角色权限（推荐）

1. 登录超管账号
2. 进入"权限控制"页面
3. 编辑角色（如"普通用户"）
4. 在权限树中勾选：
   - ✅ API管理
     - ✅ API查看
5. （可选）如果需要自定义 Scopes，在 `my:api-keys` 节点旁边的下拉框中选择：
   - `email:self` - 个人权限（默认）
   - `admin:read` - 管理员读取
   - `admin:write` - 管理员写入
6. 保存角色

#### 方式 B: 直接为单个用户启用

1. 登录超管账号
2. 进入"用户列表"页面
3. 找到目标用户
4. 点击"操作" → "API 权限"
5. 开启"允许创建 API Key"开关
6. （可选）配置该用户的最大权限范围

### 步骤 3: 用户创建 API Key

1. 用户登录后，在侧边栏"管理"部分可以看到"API Keys"菜单
2. 点击进入 API Key 管理页面
3. 点击"生成新的 API Key"按钮
4. 输入描述信息
5. 选择过期时间（可选，默认永不过期）
6. 复制生成的完整 Key（**仅显示一次**）
7. 妥善保存 Key

### 步骤 4: 使用 API Key

在 HTTP 请求头中添加：

```
X-API-Key: your-api-key-here
```

示例（使用 curl）：

```bash
# 生成新邮箱
curl -X POST https://your-domain.com/v1/emails/generate \
  -H "X-API-Key: your-api-key-here" \
  -H "Content-Type: application/json" \
  -d '{"name": "test", "domain": "example.com"}'

# 获取邮件列表
curl https://your-domain.com/v1/test@example.com/messages \
  -H "X-API-Key: your-api-key-here"

# 获取单封邮件
curl https://your-domain.com/v1/test@example.com/messages/123 \
  -H "X-API-Key: your-api-key-here"
```

## 权限模型说明

### 个人用户（默认）

```
用户属性:
  can_create_api_keys = 1
  max_api_scopes = null

创建的 API Key:
  scopes = ["email:self"]
  
权限范围:
  ✅ 只能操作自己的邮箱
  ✅ 只能查看自己的邮件
  ❌ 不能访问其他用户的资源
  ❌ 不能执行管理员操作
```

### 超管配置的用户

```
用户属性:
  can_create_api_keys = 1
  max_api_scopes = ["email:self", "admin:read"]

创建的 API Key:
  scopes = ["email:self", "admin:read"]
  
权限范围:
  ✅ 可以操作自己的邮箱和邮件
  ✅ 可以读取所有用户的数据
  ❌ 不能修改其他用户的数据（需要 admin:write）
```

### Scope 定义

| Scope | 说明 | 权限 |
|-------|------|------|
| `email:self` | 个人权限 | 仅操作自己的邮箱和邮件 |
| `admin:read` | 管理员读取 | 查看所有用户的数据 |
| `admin:write` | 管理员写入 | 修改所有用户的数据 |
| `admin` | 完全管理员 | 等同于 admin:read + admin:write |

## 安全注意事项

1. **API Key 保密**: 
   - API Key 仅在创建时显示一次
   - 请妥善保存，不要泄露给他人
   - 如果泄露，立即删除并重新生成

2. **过期时间**:
   - 建议为 API Key 设置合理的过期时间
   - 定期轮换 API Key
   - 过期的 Key 会自动失效

3. **权限最小化**:
   - 个人用户默认只有 `email:self` 权限
   - 仅在必要时授予更高权限
   - 定期审查用户的 API 权限

4. **资源所有权验证**:
   - 即使有 `email:self` 权限，后端也会验证资源所有权
   - 用户只能访问属于自己的邮箱和邮件
