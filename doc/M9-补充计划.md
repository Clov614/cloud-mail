# Cloud-Mail API 增补计划 M9 (Phase 2 补充)

## 问题分析

根据 commit `8f7b110` 和 `0b5961e`，M1-M8 已经实现了基础的 API Key 功能，但存在以下核心问题：

1. **权限控制问题**：API Key 管理被错误地放在了超管特有的权限控制中，普通用户无法使用
2. **数据库查询错误**：查询 API Key 时字段名不匹配（使用了下划线命名但数据库是驼峰命名）
3. **权限条目缺失**：权限控制页面没有显示 API Key 相关的权限条目
4. **Scopes 显示问题**：用户的 API 权限显示为 "N/A"

## 核心需求

1. **个人用户**可以在新增的侧边栏（API Key 管理）中 CRUD 自己的 API Key
2. **个人用户**的 API Key 不细分 scopes，全部 API 都能用，但**仅能操作自己的邮箱和邮件**
3. **超管**可以在权限控制中增加"API Key 管理"大条目，小条目里控制具体 scopes
4. 可以控制 API Key 的过期时间

## 关键设计决策

### 权限模型

```
个人用户 (canCreateApiKeys=1, maxApiScopes=null)
  └─ 创建的 API Key
      └─ scopes: ["email:self"]
          └─ 只能访问自己的邮箱和邮件

超管配置的用户 (canCreateApiKeys=1, maxApiScopes=["email:self","admin:read"])
  └─ 创建的 API Key
      └─ scopes: ["email:self","admin:read"]
          └─ 可以访问自己的资源 + 管理员读取权限
```

### Scope 定义

- `email:self`: 操作自己的邮箱和邮件（个人用户默认）
- `admin:read`: 管理员读取权限（查看所有用户数据）
- `admin:write`: 管理员写入权限（修改所有用户数据）
- `admin`: 完全管理员权限（等同于 admin:read + admin:write）

### 安全考虑

1. **资源所有权验证**: 即使有 `email:self` 权限，也必须验证资源属于当前用户
2. **过期检查**: 每次 API 调用都检查 Key 是否过期
3. **最小权限原则**: 个人用户默认只有 `email:self` 权限
4. **权限继承**: 用户的 `maxApiScopes` 限制了其能创建的 Key 的最大权限
